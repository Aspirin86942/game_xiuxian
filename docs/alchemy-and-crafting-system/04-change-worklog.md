- 一次规则变化记一条，不把多个独立前提混成一段流水账。

- 若兼容影响或数值影响无确切信息，明确写“需补验证”，不凭空补结论。

## 3. 设计前提

- `丹药必须是可规划资源`：当前为是；若未来退化为随机彩蛋，视为回退条件。
- `自然回血只能做保底`：当前为目标态要求；若后续超过 `50% maxHp` 或稀释回血丹价值，视为 B 级回调对象。
- `高境界突破门槛来自准备而不是纯刷脸`：当前为 issue #5 的核心前提；若被移除，必须重新立册说明。
- `旧存档兼容优先于新机制炫技`：当前为是；任何新增恢复字段或配方字段都必须先补兼容说明。

## 4. 回退条件

- 若新增规则导致旧存档无法读档，应回退。
- 若新增流程会让库存出现负数或允许缺料炼丹，应回退。
- 若自然回血与离线吐纳修为结算互相污染，应回退。
- 若新增第 6 个导航按钮后破坏 `375x667` 单屏可用范围，应回退并重审入口布局。

### `[2026-03-21] issue #5 第一阶段代码落地`

#### 改动内容

- 在 `story-data.js` 增加 `huashendan`、`ALCHEMY_RECIPES` 与自然回血参数。
- 在 `game-core.js` 落地 `craftRecipe()`、`resolveNaturalRecovery()`、`state.recovery.lastCheckedAt`、`SAVE_VERSION = 6` 与 `v5` 继续可读的兼容判断。
- 在 `game.js` / `index.html` / `style.css` 落地独立丹炉页、第 6 个底部导航按钮、战斗态切页阻断、战前丹药摘要与自然回血循环。
- 在自动化测试中补入配方成功/失败、自然回血、高阶突破丹限制与新 UI 锚点覆盖。

#### 改动原因

- 需要把规则书中的“材料 -> 配方 -> 成品丹 -> 恢复 / 突破准备”闭环变成真实运行时行为。
- 需要在不打破旧存档的前提下，引入自然回血与高阶突破丹。

#### 预期玩家体验 / 用户体验变化

- 玩家现在可以在独立丹炉页查看丹方、锁定原因与缺料信息，并直接炼制基础丹药。
- 战斗胜利后不再被固定回血抹平消耗，回血丹与自然回血重新承担续航分工。
- 元婴以下无法服用 `huashendan`，已有突破药力时也不能继续叠加突破丹。

#### 是否影响旧状态 / 旧存档 / 旧客户端

  - 新增 `state.recovery.lastCheckedAt`
