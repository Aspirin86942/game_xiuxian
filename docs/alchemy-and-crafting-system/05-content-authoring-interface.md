# 05. 内容接入接口规范

## 1. 目的

本文件规定新增材料、配方、成品丹药与剧情资源入口如何安全接入炼丹、制作与丹药系统，避免把“丹药”写成只有名字没有系统身份的装饰文本。

## 2. 一个章节最少要定义什么

若某个章节会碰到丹药系统，最少要定义：

- 资源身份：这是材料、成品丹药、稀有配方线索，还是纯叙事伏笔。
- 系统挂钩点：`effects.items`、`requirements.items`、`costs`、怪物掉落或配方解锁条件至少命中一种。
- 揭示方式：玩家在获得前是否知道这是材料或丹药，在获得后是否能在背包或丹炉页看见它。
- 失败表现：资源不足、境界不足或战斗中阻断时，系统如何明确提示。

若章节完全不碰炼丹系统，也应明确“不挂资源钩子”，避免未来误读。

## 3. 新增材料时最低字段要求

在当前代码结构下，新增材料至少应在 `story-data.js` 的 `ITEMS` 中提供：

- `name`
- `type`
- `description`
- `usable`

并至少满足以下之一：

- 作为配方成本出现在 `ALCHEMY_RECIPES[*].costs`
- 作为掉落或剧情奖励进入 `effects.items`
- 作为门槛出现在 `requirements.items`

只写文案、不进入系统读取点的材料，不算正式接入。

## 4. 新增配方时最低字段要求

每条配方至少应定义：

- `id`
- `name`
- `category`
- `summary`
- `costs`
- `outputs`
- `unlock`

并回答三个问题：

- 玩家现在花掉了什么材料。
- 玩家现在换到了什么成品丹药。
- 这个配方更偏续航、修为补足，还是关键突破准备。

## 5. 新增成品丹药时最低字段要求

新增成品丹药至少应定义：

- `ITEMS[itemId].name`
- `ITEMS[itemId].type = 'pill'`
- `description`
- `usable`
- `effect` 或 `actions`

并明确以下信息：

- 是否只能战斗外使用
- 是否受境界限制
- 是否与现有药力互斥
- 使用后是否改变修为、气血或突破加成

## 6. 何时必须写显式 `tradeoff`

出现以下情况时，必须写显式 `tradeoff`，不能只写“获得丹药”：

- 同一种材料既能直接兑现收益，又能攒作突破准备。
- 一个 choice 会把稀有材料从“未来高阶突破资源”改成“当下回血或修为”。
- 高阶突破丹的使用意味着放弃其他稀有制作路径。
- 某个剧情奖励会显著改变玩家后续突破难度或战斗续航。

显式 `tradeoff` 至少要回答：

- 你现在失去了什么材料或机会。
- 你现在换到了什么确定收益。
- 未来哪条路因此更难或更容易。

## 7. 何时只允许走默认 `tradeoff`

仅在以下情况可沿用默认 `tradeoff` 语义，不必额外扩写复杂描述：

- 普通怪物掉落少量 `lingcao`、`lingshi`、`yaodan`。
- 一个低价值章节只发放基础材料，不改变长期节点。
- 一个基础回血丹配方只消耗常见材料，不与稀有突破资源竞争。

一旦资源开始承担“现在保命 vs 未来突破”“基础炼制 vs 高阶储备”这类竞争关系，就不能再只走默认 `tradeoff`。

## 8. 什么是伪分支

以下情况视为伪分支：

- 文案里写“你炼成丹药”，但系统里没有新增配方、没有增加物品、也没有日志或 UI 证据。
- 同一材料被描述成很稀有，但它既没有掉落入口，也不进入任一配方。
- 高阶突破丹只存在于一段台词里，背包、突破率和配方系统都无法读取它。
- 一个 choice 写了“把材料留作以后”，但系统没有保存任何能影响后续读取的状态。

伪分支不得冒充正式内容接入；若只是想埋伏笔，必须明确标记为“纯叙事伏笔”。
