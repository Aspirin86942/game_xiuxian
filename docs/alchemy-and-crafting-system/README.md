# 炼丹、制作与丹药系统规则书

## 1. 目的

本目录用于定义《灵光修仙传》炼丹、制作与丹药系统的目标态规范、当前实现基线与后续落地契约，不再把关键判断只留在聊天记录里。

本册采用三层写法：

- `目标态规范`：描述 issue #5 要落成的完整闭环。
- `现状基线`：只在 `01-current-behavior.md` 中记录当前代码已做到的真实行为。
- `落地差距`：说明从当前代码走向目标态时，哪些契约必须保留，哪些临时写法必须收束。

默认原则：

> 先改规则书，再改代码。

补充说明：

- 当前运行时真实行为，仍以代码与 `01-current-behavior.md` 为准。
- 未来改代码时，设计原则、参数、契约与内容接入方式，以本册其余文件为准。

## 2. 使用顺序

1. 先看 `00-design-principles.md`，确认丹药系统为什么存在，以及为什么不能退化成“另一个按钮”。
2. 再看 `01-current-behavior.md`，确认当前已有丹药、战斗结算、背包动作和兼容边界到底做到哪一步。
3. 再看 `02-parameters-and-formulas.md`，确认本册约束的配方、恢复阈值、突破丹强度与联动风险。
4. 再看 `03-implementation-contract.md`，确认稳定字段、稳定流程、稳定 UI 契约、禁止项与最低测试。
5. 再看 `04-change-worklog.md`，确认规则变化如何留痕、何时应回退。
6. 最后看 `05-content-authoring-interface.md` 与两个附录，确认新材料、新配方、新丹药和剧情奖励如何安全接入。

## 3. 工作流

1. 先在 `01-current-behavior.md` 判断当前代码已有的“材料、背包动作、突破加成、固定战后回血”事实。
2. 再在 `00-design-principles.md` 判断这次改动是否符合“资源闭环、保底恢复、准备型突破、单屏可用”的设计宪章。
3. 再在 `02-parameters-and-formulas.md` 确认涉及的配方、恢复比例、封顶阈值、高阶突破丹强度与观察指标。
4. 再在 `03-implementation-contract.md` 判定改动属于 A/B/C 哪一级，是否触发存档兼容、UI 契约与自动化测试要求。
5. 改完规则或改完代码后，在 `04-change-worklog.md` 留下规则级变更，不把判断只留在提交信息或聊天里。
6. 若新增剧情奖励或掉落来源，回到 `05-content-authoring-interface.md` 与附录，检查是否形成真实资源闭环而不是伪分支。

## 4. 目录职责

- `README.md`：说明本册定位、边界、阅读顺序与维护规则。
- `00-design-principles.md`：定义丹药、炼制、保底回血与突破准备的设计宪章。
- `01-current-behavior.md`：只记录当前代码真实行为、兼容边界与已确认缺口。
- `02-parameters-and-formulas.md`：记录当前值、目标值、公式、联动风险与调参观测项。
- `03-implementation-contract.md`：定义稳定状态字段、稳定流程、稳定 UI 契约、禁止项、测试门槛与变更分级。
- `04-change-worklog.md`：记录规则级变更、前提变化与回退条件。
- `05-content-authoring-interface.md`：规定新增材料、配方、成品丹药与剧情资源入口的接入方式。
- `appendix-a-state-matrix.md`：收纳配方生命周期、丹药使用边界与兼容矩阵。
- `appendix-b-test-scenarios.md`：收纳设计验收场景、当前自动化覆盖与未来补测入口。

## 6. 本册定位

- 本册是系列中的第三本 mature 分册。
- 本册不是“背包系统子章节”，而是“炼丹、制作与丹药系统的独立规则书”。
- 本册保留 `01-current-behavior.md`，专门记录当前实现、兼容边界与目标态差距，不代表其余文件要被当前代码形状绑死。

## 7. 本册负责什么

- 定义材料、配方、成品丹药三类对象的系统身份与最小字段要求。
- 定义“材料 -> 配方 -> 成品丹 -> 使用/准备突破”的完整闭环。
- 定义非战斗低速保底回血的阈值、封顶与结算方式。
- 定义高境界突破丹的适用范围、禁止叠药规则与揭示方式。
- 定义独立丹炉页的最小 UI 契约、移动端单屏边界与失败提示。
- 定义丹药系统新增状态字段与旧存档兼容要求。

## 8. 本册不负责什么

- `inventory` 的基础存储模型、通用背包弹层渲染与持有型被动基础接口。
- 叙事抉择的承诺结构、回响写法、失败压力与终局语义。
- 商店、交易行、战斗内嗑药、炼丹小游戏或更重的经济系统。
- 为了丹药系统而引入新的前端框架、打包器或模块化重构。

## 9. 与其他分册的交界

- `inventory-and-item-system` 负责“物品如何存、如何显示、哪些物品可直接使用、哪些持有即生效”。
- 本册负责“材料如何炼、丹药如何来、何时能吃、恢复为什么只能保底、突破丹为何只服务准备而不服务刷脸”。
- `narrative-decision-system` 负责“为什么在剧情里给材料/扣材料/揭示资源代价”，本册负责这些资源进入系统后的持续语义。
- 当一个剧情 choice 同时涉及叙事后果与丹药资源时，choice 的叙事结构先看 `narrative-decision-system`，资源来源、配方去向和丹药效果再看本册。

## 11. 当前机制一句话摘要

- 当前实现已经具备“材料掉落 + 背包动作 + 少量丹药直用 + 突破加成”的雏形，但还没有形成独立炼丹页、稳定配方系统和非战斗保底回血闭环。

## 12. 维护规则

- 任何对目标态的修改，至少同步检查 `00`、`02`、`03`、`05` 是否仍一致。
- 任何把当前代码迁向目标态的实施，必须同步更新 `01` 与 `04`，避免差距失真。
- 任何新增材料、配方、突破丹或回血规则，若不能通过 `05` 与附录中的伪分支检查，就不应接入系统。
- 若未来要把炼丹系统扩成商店、交易或战斗内资源系统，必须先在 `04-change-worklog.md` 写清为什么当前规则不足，再升级，不在聊天里隐式漂移。
