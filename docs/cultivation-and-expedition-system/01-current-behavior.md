# 01. 当前实现行为

## 1. 机制目标

记录修炼与游历主循环系统当前已经实现的行为事实，以及从旧循环迁向新循环后保留了哪些稳定语义。

## 2. 术语表

| 中文名 | 代码名 | 当前含义 |
| --- | --- | --- |
| 修为 | `state.cultivation` | 当前小境界进度条；到达 `maxCultivation` 后才可突破 |
| 修为上限 | `state.maxCultivation` | 当前境界阶段的突破前上限 |
| 灵石 | `state.inventory.lingshi` | 当前唯一可重复修炼资源；由游历事件、剧情奖励和掉落进入背包 |
| 批量闭关 | `trainWithLingshi(state, batchKey)` | 消耗 1 / 10 / 最大可修炼枚数的灵石并换取修为 |
| 游历事件池 | `resolveExpedition(state, rng)` | 游历统一入口，负责抽取资源、战斗、损耗、线索事件 |
| 战斗事件 | `beginCombat()` / `resolveCombatRound()` | 游历事件中的一类；只负责灵石、掉落、气血变化，不再直接改修为 |
| 突破 | `attemptBreakthrough(state, rng)` | 修为满后提升 `realmIndex / stageIndex` 的唯一入口 |

## 3. 覆盖范围

当前机制覆盖：

- 游历获取灵石、掉落、风险损耗与战斗事件。
- 闭关按批次消耗灵石并增加修为。
- 修为满后突破，境界推进后继续复用既有剧情队列。
- 纯单机自由前提下的完整导出 / 导入。
- `version < 6` 存档的阻断与重置。

当前机制不覆盖：

- 离线挂机修炼。
- 自动吐纳。
- 灵石背包直接吸收转修为。
- 排行榜、成就可信边界、存档签名或反作弊。

## 4. 状态字段

- `state.version`：当前主循环版本号；`6` 起视为新循环存档。
- `state.cultivation`：当前修为值，始终保持在 `0 ~ maxCultivation`。
- `state.maxCultivation`：当前境界阶段对应的突破上限。
- `state.realmIndex` / `state.stageIndex`：当前大境界与小阶段。
- `state.inventory.lingshi`：当前唯一可重复修炼资源。
- `state.playerStats.hp` / `state.playerStats.maxHp`：游历战斗与损耗事件使用的生命状态。
- `state.storyProgress` / `state.storyCursor`：继续作为剧情推进与剧情视图的稳定入口。
- `state.ui.activeTab`：当前稳定控制“修行 / 剧情”两个主页；旧存档若写成 `adventure`，加载时会自动回落到 `cultivation`。

## 5. 结算顺序

### 5.1 闭关

1. 校验当前是否已满修为；已满则转交突破，不扣灵石。
2. 计算本次批量闭关实际可消耗的灵石数。
3. 先扣除灵石，再增加修为，且修为不得超过 `maxCultivation`。
4. 写入日志、刷新 UI、保存存档。

### 5.2 游历

1. 当修行页主按钮因灵石不足切到“出门游历”时，点击后先从事件池抽取事件。
2. 若抽中资源 / 风险 / 线索事件，直接结算资源、气血与日志。
3. 若抽中战斗事件，进入自动战斗循环。
4. 战斗结束后再回写灵石、掉落与气血，不直接回写修为。

### 5.3 突破

1. 仅在 `cultivation >= maxCultivation` 时允许突破。
2. 成功后提升境界并清空当前修为。
3. 失败后只按既有突破语义结算惩罚，不触发主循环额外扣费。
4. 突破后继续复用既有剧情与小境界事件队列。

## 6. UI 揭示

选前显示：

- 顶栏展示当前境界、当前修为、当前灵石。
- 修行页展示一个统一主按钮，其语义按“渡劫突破 / 闭关修炼 / 出门游历”三态切换。
- 修行页展示当前突破率、批量闭关按钮、当前批次消耗说明、当前地点与最近一次游历摘要。
- 剧情页展示 NPC 主动对话入口与线索 / 支线列表。
- 设置页展示“纯单机自由，完整导出/导入，不保证平衡”的提示。

选后显示：

- 闭关后显示灵石消耗与修为增长的浮字 / 日志。
- 游历事件后显示灵石得失、掉落、气血变化或战斗结果。
- 导入旧档失败时显示版本不兼容提示。

## 7. 兼容策略

- `version < 6`：不兼容；读档与导入都提示后重置为新档。
- `version >= 6`：按当前 `mergeSave()` 回填可缺省的稳定字段。
- 完整导出 / 导入仍保留，默认接受回档与改档风险；当前项目不声明可信边界。

## 8. 边界与异常

- 边界条件：
  - 灵石不足时不能进入闭关结算。
  - 修为已满时不能继续扣灵石。
  - 战斗失败与风险事件都不能让 `inventory.lingshi` 变成负数。
  - 导入旧档时不能污染当前新档状态。
- 失败路径：
  - 闭关资源不足：显式提示并保持原状态。
  - 导入旧档：显式提示版本不兼容并阻断导入。
  - 游历失败：显示损耗结果并回到安全气血，不静默失败。
