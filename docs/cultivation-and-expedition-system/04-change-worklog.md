# 04. 规则变更记录

## 1. 使用规则

本文件只记录修炼与游历主循环系统的规则变化，不记录普通代码重构或纯文案修订。

要求：

- 一次规则变化记一条，不把多个独立前提混成一段流水账。
- 必须写明为什么改、影响哪条主循环语义、是否影响旧存档、何时应该回退。
- 若兼容影响或数值影响没有确切信息，明确写“需补验证”，不凭空补结论。

## 3. 设计前提

- `游历必须是资源入口而不是固定战斗按钮`：当前为是；若未来改回“点一下必打架”，视为 A 级变更。
- `闭关必须是唯一可重复修为入口`：当前为是；若未来恢复灵石吸收、离线修炼或点击修炼，视为 A 级变更。
- `纯单机自由优先于可信平衡边界`：当前为是；保留完整导出 / 导入，不再宣称可信性。
- `旧档破坏式变更要明确版本门槛`：当前为是；本册当前门槛为 `version >= 6`。

## 4. 回退条件

- 若主循环改完后仍存在第二条可重复修为入口，应回退。
- 若游历胜负仍直接增减修为，应回退。
- 若导入旧档会污染新档状态，应回退。
- 若批量闭关会多扣灵石或修为溢出，应回退。
- 若设置页仍然暗示导出 / 导入具有可信平衡保证，应回退。

## 5. 已记录变更

### `[2026-03-21] 新建修炼与游历主循环系统成熟分册`

#### 改动内容

- 新建 `cultivation-and-expedition-system` 分册。
- 把“游历产资源、闭关消耗资源、突破推进境界、完整导出 / 导入边界”收束成第三本 mature 分册。
- 在系列索引中登记第三本 mature 分册，并明确其与 `inventory-and-item-system`、`narrative-decision-system` 的边界。

#### 改动原因

- 当前仓库原有“点击修炼 + 离线挂机 + 战斗直给修为”的主循环已经不符合新的设计目标。
- 用户明确要求先写规则书，再按规则书实施，减少后续上下文混淆。

#### 预期玩家体验 / 用户体验变化

- 主循环从“点击 / 挂机攒修为”转为“游历赚灵石 → 闭关修炼 → 突破推进剧情”。
- 设置页保留完整导出 / 导入，但明确提示当前项目是纯单机自由。

#### 是否影响旧状态 / 旧存档 / 旧客户端

- 是
- 若是，影响点：
  - `version < 6` 的旧档不再兼容，加载与导入都被阻断并重置。

### `[2026-03-21] 修炼页与游历页收敛为单页修行入口`

#### 改动内容

- 删除独立 `adventure` 页面与底部“游历”页签。
- 把当前地点、最近一次游历摘要与战斗概况收进修行主卡。
- 把 NPC 主动对话入口与线索 / 支线列表迁入剧情页。
- 把主按钮语义收敛为“出门游历 / 闭关修炼 / 渡劫突破”三态。

#### 改动原因

- 新主循环已经不再需要“修炼页”和“游历页”分开承载核心操作，双页切换会拉长主循环路径。
- 当前实现目标更接近“一个修行入口，按资源状态自动决定下一步”，因此需要同步精简 UI 和测试契约。

#### 预期玩家体验 / 用户体验变化

- 玩家默认停留在一个“修行”页中，缺灵石时直接出门游历，有灵石时直接闭关，修为满时直接突破。
- 剧情页额外承接人物关系与线索摘要，不再让主循环页同时承担过多支线入口。

#### 是否影响旧状态 / 旧存档 / 旧客户端

- 是
- 若是，影响点：
  - 旧 `version = 6` 存档若写入 `ui.activeTab = 'adventure'`，加载后会自动映射到 `cultivation`。
  - 不升存档版本，不改数值字段与资源闭环。

#### 是否需要同步更新测试

- 是
- 若是，需要更新：
  - `tests/ui-contract-smoke.js`
  - `tests/offline-smoke.js`
  - `tests/e2e/navigation.spec.js`
  - `tests/e2e/adventure.spec.js`
  - `tests/e2e/settings-save.spec.js`

#### 需要同步更新的文档

- `README.md`
