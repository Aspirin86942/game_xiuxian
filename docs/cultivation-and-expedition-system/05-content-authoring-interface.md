# 05. 内容接入接口规范

## 1. 目的

本文件规定新增游历事件、修炼文案、资源提示与导出 / 导入说明如何安全接入现有主循环，避免把“游历”“闭关”“战斗”“存档”写成只有文案没有系统语义的装饰内容。

## 2. 一个游历事件最少要定义什么

若某个游历事件接入主循环，最少要定义：

- 事件类型：`battle / resource / risk / clue`
- 结算结果：灵石变化、掉落、气血变化、是否进入战斗
- 可见反馈：玩家在事件后能看到什么
- 失败表现：资源不足、损耗或战斗失败后如何回到安全状态

若事件只是背景文本，不改变任何状态，应明确标记为“纯氛围文案”，不能冒充正式游历事件。

## 3. 一个合格 choice / 闭关入口至少要有哪三层差异

一个涉及闭关的合格交互，至少要同时具备以下三层差异：

- 资源层差异：消耗多少灵石、是否允许批量、资源不足时怎么阻断。
- 状态层差异：增加多少修为、是否触发突破、是否会碰到上限。
- 揭示层差异：玩家是否看得到当前灵石、当前批次成本、闭关后的收益。

## 4. 新增内容时最低字段要求

- 若新增游历事件定义，至少要补：
  - `id`
  - `type`
  - `title`
  - `summary`
  - `effect` 或稳定结算路径
- 若新增闭关批次，至少要补：
  - 批次键
  - 批次数量或推导公式
  - UI 标签
  - 失败提示
- 若新增设置页单机提示，必须明确：
  - 是否完整导出 / 导入
  - 是否保证平衡可信
  - 是否影响旧档兼容

## 5. 何时必须写显式 `tradeoff`

出现以下情况时，必须写显式 `tradeoff`，不能只给一句“获得 / 失去”：

- 灵石既能用于剧情支付，又会被批量闭关消耗。
- 玩家需要在继续游历攒资源和立即闭关冲境界之间做取舍。
- 战斗胜利与资源采集都能拿灵石，但风险与气血代价不同。
- 导出 / 导入提示需要明确“自由”与“不保证平衡”的边界。

## 6. 何时只允许走默认 `tradeoff`

仅在以下情况可沿用默认 `tradeoff` 语义，不必额外扩写复杂描述：

- 小额灵石得失。
- 不改变长期状态的普通资源事件。
- 不改变当前章节路线，只补一条主循环提示的线索事件。

## 7. 什么是伪分支

以下情况视为伪分支：

- 文案写着“闭关修炼”，实际仍然调用旧的点击修炼逻辑。
- 文案写着“游历探查”，实际仍然每次固定进入战斗。
- 文案写着“导出存档仅供单机参考”，但代码层仍暗示存在官方验证。
- 事件只改台词不改任何状态，却冒充正式资源 / 风险 / 线索事件。
