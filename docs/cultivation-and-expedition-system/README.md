# 修炼与游历主循环系统规则书

## 1. 目的

本目录用于定义《灵光修仙传》修炼与游历主循环的目标态规范、当前实现事实与落地契约，不再把“修为从哪里来、灵石怎么花、游历为什么存在、战斗在主循环中的位置、单机存档边界是什么”拆散在聊天记录里。

本册采用三层写法：

- `目标态规范`：描述未来应当形成的“游历产资源 → 闭关消耗资源 → 境界推进剧情”闭环。
- `现状基线`：只在 `01-current-behavior.md` 中记录当前代码真实行为、当前差距与兼容边界。
- `落地契约`：说明未来从当前实现迁移到目标态时，哪些状态、公式、UI 与测试必须被保住。

默认原则：

> 先改规则书，再改代码。

补充说明：

- 运行时真实行为，当前仍以代码与 `01-current-behavior.md` 为准。
- 未来改代码时，修炼与游历的目标设计、参数、契约与作者接口，以本册其他文件为准。

## 2. 使用顺序

1. 先看 `00-design-principles.md`，确认主循环究竟优先优化什么。
2. 再看 `01-current-behavior.md`，确认当前实现有哪些旧入口、旧耦合与现成可复用状态。
3. 再看 `02-parameters-and-formulas.md`，确认闭关收益、事件权重、失败语义与调参观测项。
4. 再看 `03-implementation-contract.md`，确认未来实施时哪些字段、流程、UI 与测试不能被静默改坏。
5. 再看 `04-change-worklog.md`，确认本册的前提变化、边界迁移与回退条件。
6. 最后看 `05-content-authoring-interface.md` 与两个附录，确认新增游历事件、训练入口、提示文案与测试场景如何接入。

## 3. 工作流

1. 先在 `00-design-principles.md` 判断这次改动是否让剧情、选择与资源投入重新回到主循环中心。
2. 再在 `01-current-behavior.md` 找当前真实行为，确认这次是在补事实、迁目标态，还是记录差距。
3. 再在 `02-parameters-and-formulas.md` 确认闭关倍率、游历产出、失败代价与观测指标。
4. 再在 `03-implementation-contract.md` 判定改动属于 A/B/C 哪一级，是否触发兼容、UI 与自动化测试要求。
5. 改完规则或改完代码后，在 `04-change-worklog.md` 留下设计留痕，不把判断只留在聊天记录里。
6. 若新增事件、按钮或剧情挂钩，回到 `05-content-authoring-interface.md` 与附录，检查字段最小集、伪分支风险与测试矩阵。

## 4. 目录职责

- `README.md`：说明本册定位、边界、阅读顺序与维护规则。
- `00-design-principles.md`：定义主循环设计宪章。
- `01-current-behavior.md`：只记录当前代码真实行为、兼容边界与已确认差距。
- `02-parameters-and-formulas.md`：记录当前可验证参数、目标态固定参数、公式、观测指标与联动风险。
- `03-implementation-contract.md`：定义稳定状态契约、稳定流程契约、稳定 UI 契约、禁止项、测试门槛与变更分级。
- `04-change-worklog.md`：记录规则级变更、边界迁移与回退条件。
- `05-content-authoring-interface.md`：规定新增游历事件、修炼文案、战斗入口与剧情挂钩如何安全接入系统。
- `appendix-a-state-matrix.md`：收纳主循环状态矩阵、游历结果矩阵、存档兼容矩阵。
- `appendix-b-test-scenarios.md`：收纳设计验收场景、当前自动化覆盖与未来补测入口。

## 6. 本册定位

- 本册是系列中的第三本 mature 分册。
- 本册不是“当前代码逐行说明书”，而是“修炼与游历主循环系统的目标规则书”。
- 本册保留 `01-current-behavior.md`，专门记录当前实现、兼容边界与目标态差距，不代表其余文件要被当前旧逻辑绑死。

## 7. 本册负责什么

- 定义游历事件池、战斗事件与资源事件如何组成主循环。
- 定义灵石如何从游历产出，如何进入闭关消耗，并最终转化为修为。
- 定义修为、突破、境界、剧情推进之间的稳定衔接关系。
- 定义“批量闭关”交互的最小可见信息、失败提示与默认参数。
- 定义纯单机自由前提下，完整存档导出/导入的文案边界与玩法承诺。

## 8. 本册不负责什么

- 物品 taxonomy、持有型被动、剧情扣费的系统主责。
- choice 承诺结构、失败压力、终局语义与分支影响命名。
- 纯剧情台词润色、角色塑造或支线文案备份。
- 为主循环重构而引入新框架、打包器或多端适配体系。

## 9. 与其他分册的交界

- `inventory-and-item-system` 负责“灵石是什么、物品如何存进背包、哪些物品有持有被动、剧情如何扣费”。
- 本册负责“灵石如何进入成长闭环、修炼入口如何消耗灵石、游历如何产出资源、战斗如何回写主循环”。
- `narrative-decision-system` 负责“剧情选择如何形成承诺、回响、失败压力与终局回指”。
- 当一个 choice 同时涉及剧情后果与主循环资源时，choice 的叙事设计先看 `narrative-decision-system`，物品身份先看 `inventory-and-item-system`，主循环节奏与闭环再看本册。

## 11. 当前机制一句话摘要

- 本册的一句话目标是：把“点击/挂机积累修为”的旧主循环，收束成“游历拿资源、闭关花资源、突破推境界、境界带剧情”的单一成长闭环。

## 12. 维护规则

- 任何对主循环目标态的修改，至少同步检查 `00`、`02`、`03`、`05` 是否仍一致。
- 任何把当前代码迁向目标态的实施，必须同步更新 `01` 与 `04`，避免差距失真。
- 任何新增游历事件、修炼按钮、失败提示或导出/导入文案，若不能通过 `05` 与附录中的接入检查，就不应接入系统。
- 若未来继续升级到下一版主循环，必须先在 `04-change-worklog.md` 写清为什么当前规则不足，再重写，不在聊天里隐式漂移。
