# 00. 设计决策准则

## 1. 目的

本文件记录《灵光修仙传》储物、灵石消费与物品作用系统的设计判断标准，而不是当前实现细节。

## 2. 系统优化优先级

- 第一优先：Correctness（正确性）——物品数量、资源结算、旧存档兼容与状态重算不能错。
- 第二优先：Maintainability（可维护性）——一种物品语义只保留一套主定义，不让剧情、背包、数值重算各自发明解释。
- 第三优先：Explainability（可解释性）——玩家拿到东西后，要能理解它是消耗品、剧情物、货币，还是持有型被动。
- 第四优先：Observability（可观测性）——资源不足、不可使用、未实现的入口，必须能从 UI、日志或测试中看出原因。

## 3. 什么叫好抉择 / 好流程 / 好规则

- 灵石必须有花法：玩家既可以把灵石留作剧情/交易成本，也应能在目标态里主动把灵石转成短期成长，而不是只会越攒越多。
- 道具必须有身份：每个物品至少要能回答“它是什么”“怎么获得”“拿到后做什么”“什么时候不能乱用”。
- 持有型被动必须可解释：像 `飞剑`、`护身法器`、`曲魂` 这类常驻加成，必须能用“持有即生效”一句话解释清楚，不能藏成隐形数值。
- 储物系统优先兼容旧存档：新规则可以扩展 item 语义，但不能轻易推翻 `inventory = { [itemId]: count }` 的基本模型。
- 一条物品链路要完整：获得、入包、可见、消耗/吸收、效果落账、状态重算、存档落地应是一条完整流水线。

## 4. 什么叫坏抉择 / 坏流程 / 坏规则

- 灵石只有进账没有出路，导致“空有灵石花不出去”长期积压。
- 同一物品既像剧情旗标、又像直接消耗品、又像隐藏被动，却没有主语义。
- 被动效果只写在状态重算里，却没有在物品文案、规则书或测试中明确说明。
- 一个物品在剧情里能扣、在背包里能点、在未来商店里能买，但三条路径的结算顺序彼此冲突。
- 为了补一个物品功能就引入装备位、额外资源表或大规模 UI 重构，超过当前系统边界。

## 5. 可接受的不透明

可以隐藏：

- 未来某个稀有物品会在第几章掉落。
- 某些剧情物的最终长期价值，前提是玩家已经知道它是“剧情物”而不是“可直接使用物”。
- 灵石吸收的最终精确转化率，前提是规则书已标记为“待定参数位”，实现前不对外宣称固定数值。

不可隐藏：

- 一个物品是否允许直接使用。
- 一个物品是否会在持有时提供常驻效果。
- 一个剧情 choice 的资源代价是否来自灵石或其他库存物。
- 一个物品被消耗后是否会改变修为、突破率、生命值或背包数量。

## 6. 高风险线定义

- 改动 `inventory` 存档结构却不同时更新 `createInitialState` 与 `mergeSave`。
- 允许库存出现负数，或让资源不足时仍能继续结算后续效果。
- 在没有规则书更新的前提下，偷偷把“持有即生效”改成“必须装备位才能生效”。
- 让灵石吸收绕过现有修为上限、突破逻辑或日志可见性。
- 在没有独立商店系统前，先在 UI 上暴露购买入口但没有对应结算与失败处理。

## 7. 系统一致性原则

- `story-data.js` 负责 item 静态定义，`game-core.js` 负责库存与效果结算，`game.js` 负责背包 UI 与操作编排；三层职责不能混写成同一套散点规则。
- 物品的主语义必须能从 `ITEMS`、规则书与测试里对应起来，不允许只在某个剧情 choice 临时起效却不声明。
- 直接使用、剧情扣费、未来灵石吸收、持有型被动应共享同一套“先验证、再结算、再重算、再持久化”的流程语言。
- 任何新增 item 若会影响 `playerStats`、突破率或剧情解锁条件，都必须同步补参数说明、实现契约与测试场景。
