# 05. 内容接入接口规范

## 1. 目的

本文件规定新增物品、剧情扣费、灵石用途与持有型被动如何安全接入现有系统，避免把物品写成只有名字没有系统含义的装饰文本。

## 2. 一个章节最少要定义什么

若某个章节会碰到物品系统，最少要定义：

- 这个章节涉及的物品身份：奖励、代价、门槛，还是纯剧情物。
- 物品挂钩点：`requirements.items`、`costs`、`effects.items` 至少命中一种，不能只在台词里提到。
- 揭示方式：玩家在选前是否知道代价，选后是否能看见获得/失去什么。
- 失败表现：资源不足时是禁用选项、阻断结算，还是延后到其他入口处理。

若章节完全不碰物品系统，也应明确“不挂物品钩子”，避免未来误读。

## 3. 一个合格 choice 至少要有哪三层差异

一个涉及物品的合格 `choice`，至少要同时具备以下三层差异：

- 资源层差异：会不会花灵石、消耗库存、要求特定物品，或获得明确物品奖励。
- 状态层差异：会不会改变修为、突破、生命、路线分、旗标或后续可达路径。
- 揭示层差异：玩家能否看见本次资源动作和结果说明，而不是只收到一句空泛台词。

只有文本口吻不同、但资源、状态、后续读取都完全相同的 choice，不算合格差异。

## 4. 新增物品时最低字段要求

在当前代码结构下，新增物品至少应在 `story-data.js` 的 `ITEMS` 中提供：

- `name`
- `type`
- `description`
- `usable`

按行为补充：

- 若是 `direct-use` 物品，必须补 `effect`，并说明使用后会改哪些状态。
- 若是 `story-cost` 物品，必须至少有一个章节通过 `costs` 或 `requirements.items` 读取它。
- 若是 `hold-passive` 物品，必须在规则书与实现中写清“持有即生效”的入口与效果，不得只在文案里暗示。
- 若是 `currency-like` 资源，必须说明它当前能消费、未来能否吸收，以及两条路径是否需要 tradeoff。

## 5. 何时必须写显式 `tradeoff`

出现以下情况时，必须写显式 `tradeoff`，不能只写“获得某物”：

- 灵石既能留作剧情/交易支付，又能在目标态里被主动吸收成长。
- 一个 choice 会把同一种资源从“保留未来用途”改成“立即兑现收益”。
- 一个特殊物品会提供持续被动，但拿它意味着放弃其他路线、关系或成本控制。
- 某个消耗品使用后会显著改变突破、战斗或剧情门槛。

显式 `tradeoff` 至少要回答：

- 你现在失去了什么。
- 你现在换到了什么。
- 未来哪些路因此会更难或更容易。

## 6. 何时只允许走默认 `tradeoff`

仅在以下情况可沿用默认 `tradeoff` 语义，不必额外扩写复杂描述：

- 小境界事件里的低价值材料掉落或少量灵石得失。
- 不改变长期状态、只改变一次性库存的普通奖励。
- 对应资源没有第二条竞争用途，不会让玩家误判“该留还是该花”。

一旦资源开始承担“消费 vs 吸收”“直用 vs 留作门槛”“持有即生效 vs 卖掉换钱”这类竞争关系，就不能再只走默认 `tradeoff`。

## 7. 什么是伪分支

以下情况视为伪分支：

- 两个 choice 文案不同，但都不给/不扣任何物品，也不改变后续读取状态。
- 章节里写了“买东西”“吸收灵石”“得到异宝”，但代码层没有 `costs`、`effects.items` 或其他稳定读取点。
- 一个特殊物品被写得像会长期生效，但系统里没有任何数值、状态或 UI 证据支撑它。
- 物品只在获得时出现一次名字，之后既不能显示、不能使用、不能扣费、也不会触发后续读取。

伪分支不得冒充正式内容接入；若只是想埋伏笔，必须明确标记为“纯叙事伏笔”，不能占用物品系统名义。
