# 储物、灵石消费与物品作用系统规则书

## 1. 目的

本目录用于定义《灵光修仙传》储物、灵石消费与物品作用系统的目标态规范，而不是继续把当前背包实现拆成零散聊天结论。

本册采用三层写法：

- `目标态规范`：描述未来应当形成的储物、消费、直接使用与持有型被动闭环。
- `现状基线`：只在 `01-current-behavior.md` 中记录当前实现、关键缺口与兼容边界。
- `落地差距`：说明从当前代码走向目标态时，哪些语义必须被保留，哪些临时写法应被收束。

默认原则：

> 先改规则书，再改代码。

补充说明：

- 当前运行时真实行为，仍以代码与 `01-current-behavior.md` 为准。
- 未来改代码时，设计原则、参数、契约与内容接入方式，以本册其他文件为准。

## 2. 使用顺序

1. 先看 `00-design-principles.md`，确认灵石、道具、被动效果和旧存档兼容的优先级。
2. 再看 `01-current-behavior.md`，确认当前已有的背包、剧情扣费、可用丹药与硬编码持有被动到底做到了哪一步。
3. 再看 `02-parameters-and-formulas.md`，确认当前可验证参数、待定参数位、公式与联动风险。
4. 再看 `03-implementation-contract.md`，确认未来实施时哪些字段、流程顺序、UI 契约和最低测试不能被静默改坏。
5. 再看 `04-change-worklog.md`，确认规则变更如何留痕、何时应回退。
6. 最后看 `05-content-authoring-interface.md` 与两个附录，确认新增物品、剧情扣费、灵石吸收和持有型被动如何安全接入。

## 3. 工作流

1. 先在 `00-design-principles.md` 判断这次改动是否符合“灵石必须有花法、道具必须有身份、被动必须可解释、旧存档优先兼容”。
2. 再在 `01-current-behavior.md` 找当前真实行为，判断这次是补事实、补目标态，还是记录差距。
3. 再在 `02-parameters-and-formulas.md` 确认涉及的消耗、效果、被动强度、堆叠规则与待定参数位。
4. 再在 `03-implementation-contract.md` 判定改动属于 A/B/C 哪一级，是否触发兼容、UI 与自动化测试要求。
5. 改完规则或改完代码后，在 `04-change-worklog.md` 留下设计留痕，不把判断只留在聊天记录里。
6. 若新增物品内容或剧情挂钩，回到 `05-content-authoring-interface.md` 与附录，检查字段最小集、tradeoff 和伪分支风险。

## 4. 目录职责

- `README.md`：说明本册定位、边界、阅读顺序与维护规则。
- `00-design-principles.md`：定义储物、货币、直接使用与持有型被动的设计宪章。
- `01-current-behavior.md`：只记录当前代码真实行为、兼容边界与已确认缺口。
- `02-parameters-and-formulas.md`：记录当前可验证参数、目标态待定参数位、公式、观测指标与联动项。
- `03-implementation-contract.md`：定义稳定数据契约、稳定流程契约、稳定 UI 契约、禁止项、测试门槛与变更分级。
- `04-change-worklog.md`：记录规则级变更、前提变化与回退条件。
- `05-content-authoring-interface.md`：规定新增物品、剧情扣费、直接使用与持有型被动的内容接入方式。
- `appendix-a-state-matrix.md`：收纳物品生命周期、灵石行为与存档兼容矩阵。
- `appendix-b-test-scenarios.md`：收纳设计验收场景、当前自动化覆盖与未来补测入口。

## 6. 本册定位

- 本册是系列中的第二本 mature 分册。
- 本册不是“当前背包代码逐行说明书”，而是“储物、灵石消费与物品作用系统的目标规则书”。
- 本册保留 `01-current-behavior.md`，专门用于记录当前实现、兼容边界与目标态差距，不代表其余文件要被当前代码形状绑死。

## 7. 本册负责什么

- 定义物品的系统身份：`quest / material / pill / passive / currency-like` 的职责分工与解释方式。
- 定义灵石的系统语义：既可用于剧情/交易消费，也可以在目标态里作为主动吸收资源。
- 定义 `direct-use / story-cost / absorb / hold-passive` 四类行为如何结算、何时揭示、何时禁止。
- 定义 `inventory` 存档模型、状态重算与背包 UI 的稳定接口，不引入装备位。
- 定义新增物品与剧情扣费内容的最低接入门槛、最小测试要求和伪分支判定。

## 8. 本册不负责什么

- 炼丹、制作、合成、装备位或法宝栏位系统。
- 叙事 choice 的承诺结构、回响写法、失败压力和终局语义。
- 纯数值平衡之外的章节台词润色与剧情备份。
- 为了物品系统而引入新的前端框架、打包器或 UI 架构。

## 9. 与其他分册的交界

- `narrative-decision-system` 负责“为什么在剧情里给/扣某个物品、这次选择的后果如何叙事化揭示”。
- 本册负责“这个物品是什么、拿到之后能做什么、何时能花、何时能吸收、何时持有即生效”。
- 当一个 choice 同时涉及叙事后果与物品代价时，choice 的叙事设计先看 `narrative-decision-system`，物品代价与结算语义再看本册。
- 未来若拆出 `alchemy-and-crafting-system`，本册只保留制作品作为输入/输出资源的接口，不继续承接制作配方本体。

## 11. 当前机制一句话摘要

- 当前实现已经具备“背包 + 剧情扣费 + 丹药直接使用 + 少量硬编码持有被动”的连续链路，但仍缺“灵石主动吸收”和“独立商店/交易闭环”。

## 12. 维护规则

- 任何对目标态的修改，至少同步检查 `00`、`02`、`03`、`05` 是否仍一致。
- 任何把当前代码迁向目标态的实施，必须同步更新 `01` 与 `04`，避免差距失真。
- 任何新增 item、价格、掉落或被动效果，若不能通过 `05` 与附录中的伪分支检查，就不应接入系统。
- 若未来升级到 v2 以上，必须先在 `04-change-worklog.md` 写清为什么当前规则不足，再重写，不在聊天里隐式漂移。
