- 哪些旧实现约束只是过渡事实，不再视为长期硬约束
- 哪些流程顺序、可见性和测试门槛不能被偷换

## 3. 稳定数据契约

### 3.1 决策输入层

未来实现必须能稳定表达：

- `承诺类型`
- `可见成本或机会成本`
- `风险语义档位`
- `可支付性或可执行性`

旧实现中只有按钮文案与 `costs` 的情况，不再视为充分契约。

### 3.2 后果输出层

未来实现必须能稳定表达：

- `branchImpact.title`
- `branchImpact.detail`
- `branchImpact.promiseLabel`
- `branchImpact.riskLabel`
- 当前失败压力区间与趋势
- 至少一个隐藏承接读取点或终局种子

旧实现中的 `immediateResult`、`longTermHint`、`recentChoiceOutcome` 可以作为过渡来源，但不要求永久沿用其字段形状，也不应继续作为剧情页主显示契约。

### 3.3 分支记忆层

未来实现必须有稳定入口承接：

- 主线推进记忆
- 路线人格权重
- 关系承接
- 长期失败压力
- 终局种子
- 可回看的分支影响历史

旧实现里的 `storyProgress`、`routeScores`、`npcRelations`、`storyConsequences`、`flags` 可作为迁移素材，但不强制保持原字段名。

`pendingEchoes`、`chapterChoices`、`recentChoiceEcho` 只应作为兼容层，不应重新变回剧情页主显示数据源。

### 3.4 揭示契约层

未来实现必须能区分：

- 选前必须揭示的信息
- 选后立即可见的单条分支影响
- 选后立即可见的失败压力区间与趋势
- 延迟兑现但不直接并列展示的隐藏承接
- 终局回指的信息

若系统只保留“选前按钮”和“选后数值”，或重新回到“多卡并列复述同一后果”，则不算完成 v2。

## 4. 稳定流程契约

### 4.1 决策流程顺序

未来实现必须遵守的最低顺序：

1. 先构造决策输入层，确保承诺、成本、风险语义可用。
2. 玩家做出选择后，先结算资源与状态变化，并更新失败压力。
3. 再写入分支记忆与分支影响历史。
4. 再注册隐藏承接读取点或终局种子。
5. 最后根据累计压力与关键承诺链决定是否进入终局或下一阶段。

### 4.2 解释流程顺序

未来实现必须保证：

1. 玩家先看到“刚才做了什么承诺”留下的单条分支影响。
2. 玩家同时看到当前失败压力所在区间与趋势。
3. 后续章节或终局再去读取隐藏承接，而不是把同义解释立刻拆成多张卡并列展示。

不允许把终局解释写成与此前选择脱节的独立结论。

### 4.3 当前旧约束中被明确放弃的部分

- 不再把“选后固定展示即时结果 + 长期提示 + 到期延迟回响”视为目标态硬约束。
- 不再把 `pendingEchoes` 直接推入剧情页 `echo-list` 视为目标态能力。
- 不再把同承诺类型共用一套标题或模板正文视为可接受内容产出方式。
- 不再把 `battleWill / tribulation` 二元结构视为唯一长期后果模型。

若未来实施继续沿用旧字段，必须说明这是过渡方案而不是最终设计。

## 5. 稳定 UI 契约

### 5.1 选前必须可见

- choice 的承诺类型
- 可见成本或机会成本
- 风险语义档位
- 不可执行时的阻断原因

### 5.2 选前必须隐藏

- 精确收益数值
- 精确失败压力数值
- 隐藏承接的具体触发章节
- 终局种子的内部计算细节

### 5.3 选后必须揭示

- 恰好 1 条分支影响卡
- 当前失败压力所处区间与趋势说明
- 分支影响的轻量元信息，用于回看“你之前选了什么”

以下内容不得再作为剧情页主显示卡片并列出现：

- 即时结果摘要
- 长期提示摘要
- 旧 `pendingEchoes` 余波卡

### 5.4 终局页必须保留

- 最终分支影响链的回指说明
- 失败或成功的主要成因
- 重开或继续探索其他路线的入口

## 6. 禁止项

1. 不能一次选择生成多张只是在重复同一含义的可见卡片。
2. 不能只新增按钮文案，不新增系统后果。
3. 不能只新增隐藏状态，不新增后续读取点。
4. 不能把关键风险完全隐藏，只靠事后解释补锅。
5. 不能把失败写成随机惩罚，而不回指长期压力与关键承诺链。
6. 不能让同承诺类型的 choice 复用同一标题 / 正文模板，导致历史列表失去辨识度。
7. 不能只改代码不改 `01` 与 `04`，导致现状差距失真。

## 7. 可改项

- 承诺类型的具体枚举名
- 风险语义的具体文案
- 隐藏承接兑现窗口的长度
- 分支影响元信息的排布方式
- 失败压力分层名称
- 终局种子回指深度

只要这些调整影响玩家理解、作者接口或审计口径，就必须同步更新 `02` 与 `04`。

## 9. 变更分级

### 9.1 A 级：规则重定义

包括但不限于：

- 改动风险揭示原则
- 改动可见后果结构（例如从单条分支影响改回多卡）
- 改动失败压力分层逻辑
- 改动终局回指结构
- 改动“伪分支”的判定标准

要求：

- 先改规则书
- 必须更新 `02`、`03`、`04`
- 必须新增或更新对应审计场景

### 9.2 B 级：系统调参

包括但不限于：

- 调整风险档位数量
- 调整隐藏承接窗口
- 调整分支历史显示上限
- 调整终局种子回指深度

要求：

- 必须更新 `02`
- 必须记录预期体验变化
- 必须明确是否影响现有落地方案

### 9.3 C 级：内容扩写

包括但不限于：

- 新增章节
- 新增 choice
- 新增分支影响
- 新增隐藏承接
- 新增终局种子

要求：

- 通过 `05` 的作者接口检查
- 通过附录中的伪分支与承接审计

## 10. 设计验收

### 10.1 玩家可解释性审计

- 玩家能否说出自己做了什么承诺。
- 玩家能否从分支影响中说出自己留下了什么后果。
- 玩家能否理解当前压力为何会升高或缓和。

### 10.2 设计一致性审计

- 选前揭示是否充分但不过度透题。
- 选后结果是否收敛成单条分支影响，而不是多卡复述。
- 高风险是否来自压力累积，而不是随机暴毙。

### 10.3 资产接入质量审计

- 新内容是否真的写入分支记忆。
- 新内容是否真的具备后续读取点。
- 新内容是否通过伪分支检查。
- 新内容是否没有过度依赖兜底生成器。
