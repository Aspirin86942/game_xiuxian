# 05. 内容接入接口规范

## 1. 目的

本文件定义理想化叙事决策系统 v2 的内容作者接口，确保章节、choice、延迟回响与终局种子都能以统一方式接入系统。

内容接入不是纯写作任务，而是“叙事资产 + 系统接口”的联合设计。

## 2. 一个章节最少要定义什么

### 2.1 主线章节

一个合格的主线章节至少要定义：

- `章节目标`
- `进入条件`
- `章节摘要`
- `beats`
- `choices`
- `本章至少一个后续读取点`

每章还必须明确：

- 本章主要承接哪类历史决策
- 本章准备写入哪些新的分支记忆
- 本章是否产生延迟回响候选或终局种子

### 2.2 支线 / 悟境 / 特殊事件章节

一个合格的支线或悟境事件至少要定义：

- `触发条件`
- `独立叙事意义`
- `choices`
- `至少两层系统痕迹`

说明：

- 即便它不推进主线，也必须改变路线人格、关系承接、失败压力、回响或终局种子中的至少两项。
- 只提供插曲文本、不写系统后果的内容，不应进入成熟内容池。

## 3. 一个合格 choice 至少要有哪三层差异

### 3.1 表层差异：决策承诺

- 玩家必须能从按钮文案与上下文中理解：这次 choice 到底在承诺什么。
- 承诺必须可区分，例如偏保全、偏献祭、偏试探、偏决裂、偏夺取。

### 3.2 中层差异：可见成本 / 风险语义

- 玩家至少要看到一种成本或机会成本。
- 玩家至少要看到一种风险语义，不要求精确值，但要知道自己是否在涉险。

### 3.3 深层差异：后果承接

- choice 至少要在以下项目中命中两项：
  - 即时结果
  - 分支记忆写入
  - 延迟回响候选
  - 终局种子
  - 失败压力变化

若只有表层差异，没有中层和深层差异，默认视为伪分支。

## 4. 何时必须写显式 `tradeoff`

以下情况必须显式声明 `tradeoff`，不能只靠默认推导：

- 关键剧情转折
- 违背路线默认气质的逆向选择
- 明确承担高风险高收益语义的 choice
- 会显著推升失败压力或终局种子的 choice
- 需要把关系收益与压力代价同时绑定的 complex choice

说明：

- 在 v2 里，显式 `tradeoff` 不只是数值包，更是隐藏后果包的作者声明入口。

## 5. 何时只允许走默认 `tradeoff`

以下情况应优先使用默认 `tradeoff`：

- 常规承接章节中的普通分流
- 不承担核心世界观转折的日常 choice
- 只是延续既有路线人格的稳定 choice

默认 `tradeoff` 的作用是保持系统一致性，而不是替代关键 choice 的个性化设计。

## 6. 什么是伪分支

以下情况默认属于伪分支：

- 按钮文案不同，但承诺类型、风险语义与读取点完全相同。
- 只有数值差，没有关系、路线、回响或终局承接差异。
- 写了分支记忆，却没有后续读取点。
- 终局说明无法回指该 choice 的存在。
- 需要作者自己记住“以后再补”，但规则书中没有登记读取点。

## 7. 作者最小接入清单

新增章节或 choice 前，至少检查：

1. 本次决策承诺是什么。
2. 玩家选前能看到什么成本和风险语义。
3. 玩家选后立刻会理解什么结果。
4. 本次写入哪些分支记忆。
5. 这些记忆将在哪些后续点被读取。
6. 本次是否产生延迟回响候选或终局种子。
7. 本次是否会提升失败压力。
8. 是否能通过伪分支检查。

## 8. 内容扩写的设计底线

- 不允许只扩文本，不扩后果结构。
- 不允许只扩隐藏状态，不扩读取点。
- 不允许把高风险语义伪装成普通选择。
- 不允许把关键失败写成“作者心里知道、规则书里没写”的隐规则。
- 不允许让章节作者依赖代码细节才能判断 choice 是否合格。
