# 05. 内容接入接口规范

## 1. 目的

本文件规定剧情资产如何接入当前叙事决策系统，避免新增章节、悟境事件和 choice 变成“看起来很多，实际上不进入系统”的伪内容。

## 2. 一个章节最少要定义什么

### 2.1 主线章节

一个主线章节至少应定义：

- `id`
- `title`
- `summary`
- `requirements`
- `beats`
- `choices`

每个章节的 choice 至少应在以下能力中选择其一：

- `nextChapterId`
- `ending`
- `effects.routeScores`
- `effects.flags`
- `effects.relations`
- 显式 `tradeoff`

### 2.2 小境界事件

一个小境界事件至少应定义：

- `id`
- `realmScore`
- `title`
- `summary`
- `requirements`
- `beats`
- `choices`

小境界事件虽然不写 `chapterChoices`，但必须能通过 choice 写入系统中的至少两层痕迹。

## 3. 一个合格 choice 至少要有哪三层差异

一个设计合格的 choice，至少应同时体现三层差异：

### 3.1 表层立场差异

- 玩家在按钮文案和剧情语气上能区分它的立场。

### 3.2 中层系统差异

- 它至少改变资源、关系、路线分值、flags、`tradeoff` 之一。

### 3.3 深层承接差异

- 它要么影响后续回响，要么影响后续章节 / ending 条件，要么改变长期风险收益风格。

如果只满足表层立场差异，不满足中层和深层差异，默认视为伪分支。

## 4. 何时必须写显式 `tradeoff`

以下情况必须优先考虑显式 `tradeoff`，不能只靠默认三档语义：

- 关键剧情
- 违背默认三档气质的 choice
- 终局前高风险 choice
- 小境界中承担世界观或道心转折的悟境选项
- 需要明显高于同章其他选项的风险 / 收益差异时

## 5. 何时只允许走默认 `tradeoff`

以下情况默认应走当前系统的默认 `tradeoff` 推导，而不是手写特例：

- 普通过渡 choice
- 气质清晰但不承担关键平衡职责的 choice
- 只是为了维持路线一致性的常规承接节点

手写显式 `tradeoff` 不是装饰，它意味着这个 choice 在系统里承担了额外平衡职责。

## 6. 什么是伪分支

以下情况默认属于伪分支：

- 写了两个按钮，但 `nextChapterId`、flags、后续回响和长期承接几乎一样。
- 文案立场相反，但系统层只有非常轻微的数值差。
- 写了 `routeScores`，却没有任何后续读取点。
- 写了显式 `tradeoff`，但玩家后续根本感受不到差异。

## 7. 内容接入最小检查单

新增章节或事件前，至少检查：

1. 这个章节写入了哪些系统状态。
2. 这些状态在哪些后续点会被读取。
3. choice 是否具备表层 / 中层 / 深层三层差异。
4. 是否需要显式 `tradeoff`。
5. 是否会构成过强的单章优势解。

## 8. 内容扩写的设计底线

- 新内容不能破坏“选前隐藏、选后揭示”。
- 新内容不能只靠文本张力掩盖系统同质化。
- 新内容如果引入新 flag，必须能说明后续读取点。
- 新内容如果影响高风险线，应在 `04-change-worklog.md` 记录体验预期与回退条件。
