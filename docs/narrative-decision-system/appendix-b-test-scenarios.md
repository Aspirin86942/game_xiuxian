# Appendix B. 测试场景与设计审计

## 1. 目的

本附录用于验证理想化叙事决策系统 v2 是否真的具备以下能力：

- 选前能理解承诺与风险
- 选后只看到 1 条可回看的分支影响
- 隐藏承接能在后面重新被认出
- 失败压力可管理
- 终局能回指关键承诺链

## 2. 目标态验收场景

### 2.1 清晰承诺场景

- 目标：验证玩家在选前能区分 choice 的承诺类型。
- 检查点：
  - 按钮文案是否体现承诺差异
  - 风险语义是否可见
  - 不依赖精确数值也能做出理解性的判断

### 2.2 单条分支影响场景

- 目标：验证玩家在选后只会获得 1 条可见分支影响，而不是多张同义卡片。
- 检查点：
  - 是否只出现 1 张分支影响卡
  - 标题是否具有独立主题，而不是承诺标签换皮
  - 正文是否既能回指当前感受，又能暗示它会继续留下来
  - 元信息是否足以回看“你之前选了什么”

### 2.3 隐藏承接场景

- 目标：验证 2~5 个章节单元后，玩家仍能认出某次关键选择的承接。
- 检查点：
  - 隐藏承接是否能指回原始 choice
  - 是否不是空洞致敬文本
  - 是否真的影响后续关系、路径或终局判断
  - 是否没有在剧情页上提前堆成重复可见卡片

### 2.4 高风险压线场景

- 目标：验证高风险路线不是随机暴毙，而是可管理的压力玩法。
- 检查点：
  - 风险区间是否能被玩家感知
  - 是否存在压线前进空间
  - 若失败，说明是否能回指多次分支影响累积

### 2.5 终局解释场景

- 目标：验证终局不是孤立页面，而是完整因果链的回收。
- 检查点：
  - 是否解释最终为何成功 / 失败
  - 是否回指最近关键分支影响链
  - 是否能区分“最后一击”与“长期累积”

### 2.6 终局卷分流场景

- 目标：验证改编第五卷《归乡飞升》的终局分流不是临时菜单，而是长期路线、关系、旧账与压力边界的回收。
- 检查点：
  - `24 / 24_old_debt_and_name / 24_bond_destination / 25 / 25_final_branch` 是否形成稳定 5 章链路
  - 五条主终局是否都能回指前几卷的关键关系与承诺链
  - 异常失败终局 `zouhuo_rumo` 是否会优先抢占，而不是和主终局并列显示
  - 终局候选是否只显示当前状态真正符合条件的结果

### 2.7 玩家可见禁词烟雾场景

- 目标：验证剧情页、旧事区、卷末标题与终局摘要不会重新暴露技术黑话或产品化元语言。
- 检查点：
  - 玩家可见文本中不应直接出现：`分支影响`、`支线任务`、`卷末收束`、`终局分流`
  - 玩家可见文本中不应直接出现：`关键旗标`、`固定菜单`、`固定三选一`、`文字标签`、`资源兑换`、`情绪奖励`、`关系后果`
  - 对应位置应使用：`因果回响`、`旧事 / 旧事线索`、`此卷尽处`、`门前问心`
  - 回响、卷末、终局正文应保持修仙世界内口径，不得直接拼贴《凡人修仙传》原句

## 3. 设计审计入口

### 3.1 玩家可解释性审计

至少检查：

1. 玩家能否说出刚才选择的承诺类型。
2. 玩家能否复述分支影响标题和这一步留下的主要后果。
3. 玩家能否理解为什么当前压力会升高或缓和。

### 3.2 分支有效性审计

至少检查：

1. 同承诺类型的不同 choice 是否拥有不同分支影响主题。
2. 是否存在明确的后续读取点。
3. 是否不是“只改文案不改系统”的伪分支。
4. 是否没有继续复用系统化模板句。

### 3.3 失败可管理性审计

至少检查：

1. 高风险是否来自长期压力而不是随机事件。
2. 玩家是否能识别自己接近失控。
3. 终局是否能解释为长期选择积累。

### 3.4 玩家词面审计

至少检查：

1. 玩家是否只能看到世界观内词面，而不是实现术语。
2. 旧事区、回响区、卷末章标题、终局章标题的命名是否前后一致。
3. 文案是否做到“借气质不借原句”，而不是直接改写离线参考文本。

## 4. 文档一致性审计

每次规则书大改后，至少检查：

1. `README.md`、`00`、`02`、`03`、`05` 对同一术语的定义是否一致。
2. `01` 的现状偏差是否能对应 `03` 的目标契约。
3. `04` 的变更记录是否解释了为什么要改写目标态。
4. `01` 中提到的兼容字段是否都已在 `03` 标成兼容层而非主契约。

## 5. 迁移验证清单

若未来按 v2 改代码，至少验证：

1. 当前旧字段是否已被正确映射到目标态语义。
2. 旧存档或旧 UI 锚点若被放弃，是否先在文档中登记。
3. 旧实现中的 `pendingEchoes` 是否已退为隐藏兼容层，而不是继续被拿来堆可见卡片。
4. 旧历史缺失 `branchImpact` 时，兼容兜底是否能生成可显示且不重复的分支影响。
5. 新增章节是否已通过伪分支检查。
6. 终局卷是否把正常主终局与异常失败终局写成了可审计的独立边界。
