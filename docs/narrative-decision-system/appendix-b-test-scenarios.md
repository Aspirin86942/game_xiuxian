# Appendix B. 测试场景与设计审计

## 1. 目的

本附录用于验证理想化叙事决策系统 v2 是否真的具备以下能力：

- 选前能理解承诺与风险
- 选后能解释即时结果
- 延迟后果能重新被认出
- 失败压力可管理
- 终局能回指关键承诺链

## 2. 目标态验收场景

### 2.1 清晰承诺场景

- 目标：验证玩家在选前能区分 choice 的承诺类型。
- 检查点：
  - 按钮文案是否体现承诺差异
  - 风险语义是否可见
  - 不依赖精确数值也能做出理解性的判断

### 2.2 即时结果场景

- 目标：验证玩家在选后能立刻理解结果，而不是只看到隐藏状态变化。
- 检查点：
  - 是否出现即时结果摘要
  - 是否能回指刚才的承诺
  - 是否能识别风险上升还是缓和

### 2.3 延迟回响场景

- 目标：验证 2~5 个章节单元后，玩家仍能认出某次关键选择的承接。
- 检查点：
  - 延迟回响是否能指回原始 choice
  - 是否不是空洞致敬文本
  - 是否真的影响后续关系、路径或终局判断

### 2.4 高风险压线场景

- 目标：验证高风险路线不是随机暴毙，而是可管理的压力玩法。
- 检查点：
  - 风险区间是否能被玩家感知
  - 是否存在压线前进空间
  - 若失败，说明是否能回指多次 choice 累积

### 2.5 终局解释场景

- 目标：验证终局不是孤立页面，而是完整因果链的回收。
- 检查点：
  - 是否解释最终为何成功 / 失败
  - 是否回指最近关键承诺链
  - 是否能区分“最后一击”与“长期累积”

## 3. 设计审计入口

### 3.1 玩家可解释性审计

至少检查：

1. 玩家能否说出刚才选择的承诺类型。
2. 玩家能否说出这次选择的主要风险。
3. 玩家能否理解为什么会得到当前结果。

### 3.2 分支有效性审计

至少检查：

1. choice 是否具备两层以上后果。
2. 是否存在明确的后续读取点。
3. 是否不是“只改文案不改系统”的伪分支。

### 3.3 失败可管理性审计

至少检查：

1. 高风险是否来自长期压力而不是随机事件。
2. 玩家是否能识别自己接近失控。
3. 终局是否能解释为长期选择积累。

## 4. 文档一致性审计

每次规则书大改后，至少检查：

1. `README.md`、`00`、`02`、`03`、`05` 对同一术语的定义是否一致。
2. `01` 的现状偏差是否能对应 `03` 的目标契约。
3. `04` 的变更记录是否解释了为什么要改写目标态。

## 5. 迁移验证清单

若未来按 v2 改代码，至少验证：

1. 当前旧字段是否已被正确映射到目标态语义。
2. 旧存档或旧 UI 锚点若被放弃，是否先在文档中登记。
3. 旧实现中“只保最近一次”的结构，是否已被显式替换或明确保留为过渡方案。
4. 新增章节是否已通过伪分支检查。
