# 00. 设计原则

## 1. 目的

用一组明确原则约束支线任务系统的后续设计与实现判断，避免支线继续变成“有标题、有文案、没收尾”的装饰面板。

## 2. 系统优化优先级

### 2.1 第一优先：支线服务卷主题

- 支线首先要回答“它在补这卷的哪一笔账”，而不是“这里还能再加一条任务”。
- 支线可以扩展主线厚度，但不能抢走主线卷末出口。
- 若“多塞一条支线”和“先把卷收住”冲突，优先收卷。

### 2.2 第二优先：生命周期可解释

- 玩家必须能理解支线为什么出现、为什么会错过、为什么会失败。
- 玩家必须能理解任务在卷末是怎么被解释掉的。

### 2.3 第三优先：作者接口稳定

- 内容作者必须能用统一字段写清：
  - 它属于哪一卷
  - 它补哪种旧账
  - 它在卷末怎么收
  - 它会给后续留下什么读取点

### 2.4 第四优先：不抢主线播放权

- 主线优先推进。
- 支线只允许补账、补人情、补余波、补立场，不允许在关键主线窗口直接夺走剧情中心。

### 2.5 第五优先：奖励与代价一致

- 奖励强度必须和主线窗口、境界阶段、风险语义一致。
- 不能用高奖励掩盖支线本身没有后果承接的问题。

## 3. 什么叫好支线

一个好的支线至少满足：

1. 它能被一句话归类成某种卷主题，例如旧情、旧案、门内人情、守门、断尾。
2. 玩家知道它为什么在这个窗口出现。
3. 玩家知道它失败、错过或卷末被回收后分别意味着什么。
4. 它至少改变两层系统痕迹，例如关系 + 读取点，或奖励 + 后续线索。
5. 它的结局能在卷末被解释，而不是继续浮在界面上。

## 3.1 什么叫好抉择

- 玩家知道自己是在补账、护人、封口、回信还是切割。
- 抉择至少改变两层系统痕迹，而不是只有奖励数字差异。
- 抉择要帮助当前卷更清楚，而不是让卷末更散。

## 4. 什么叫坏支线

- 只换按钮文案，不换触发条件、奖励结构和后续读取点。
- 只给灵石或道具，不写人物、名声、旧账或后续回响。
- 把原著桥段直接改写成插曲文本，却不适配当前项目口径。
- 支线进入卷末仍然以 `available / active` 形式悬挂。
- 支线明明补的是本卷旧账，却把责任全部推给下一卷。

## 4.1 什么叫坏抉择

- 文案不同，但结果态和 `followupHook` 完全相同。
- 只有资源差异，没有卷末解释差异。
- 玩家在卷末还不知道自己这条支线到底算做了什么。

## 5. 合法的卷末去向

支线在卷末只允许进入以下三类去向：

- `volume_close`：卷内结清，不再跨卷。
- `seed_forward`：卷内结算，但保留少量后续读取点。
- `convert_to_main`：在卷末主线高潮或尾声里被主线回收解释。

不允许的去向：

- 继续 `available`
- 继续 `active`
- 靠作者记忆“以后再补”

## 6. 可接受的不透明

- 可以不在选前显示精确奖励数值，但必须让玩家知道偏人情、偏资源还是偏断尾。
- 可以不公开具体读取章节，但必须让玩家知道任务会留下后续影响。
- 可以保留隐藏承接，但不能把失败或错过条件全部藏起来。

## 7. 高风险

以下改动默认高风险：

- 改变任务状态枚举、窗口或卷末处理语义
- 让多个支线在无冲突策略的前提下同时拥挤到卷末
- 把唯一奖励设计成可重复刷取
- 把原著中不适配当前项目的桥段直接导入任务池
- 用新字段写入正式存档却不补兼容层

## 8. 设计判断捷径

未来判断一条支线是否值得落地，优先问六个问题：

1. 它补的是哪一卷的哪笔账。
2. 它为什么出现在这个主线窗口。
3. 它在卷末会怎样被解释。
4. 玩家为什么会接、会犹豫、会错过。
5. 它会留下哪些后续读取点。
6. 如果删掉它，本卷是否会更清楚，还是会失去必须的厚度。

## 9. 系统一致性

- 主线窗口、支线窗口、卷末处理与后续读取点必须相互对得上。
- 已经声明 `volume_close` 的支线，不应在下一卷继续以未完成状态出现。
- 已经声明 `convert_to_main` 的支线，卷末必须有主线读取动作与解释。
