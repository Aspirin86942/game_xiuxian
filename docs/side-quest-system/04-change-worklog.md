# 04. 变更记录

## 1. 使用规则

- 每次修改支线任务系统的规则、阈值、状态流转、冲突策略或内容接入方式后，都要在本文件记录。
- 本文件不是 Git 提交历史；重点记录“为什么改”“改动前提是什么”“什么情况下应回退”。
- 若本轮只完成文档、尚未落代码，也要明确写出“当前实现仍停留在线索层”。

## 2. 记录模板

```md
### YYYY-MM-DD
- 改动内容：
- 改动原因：
- 预期体验变化：
- 兼容影响：
- 测试影响：
- 回退条件：
```

## 3. 设计前提

- 当前代码已经同时存在 formal quest v1 与 legacy 显性支线线索，两者共用同行回响区。
- 原著全文只作为离线参考素材，不是运行时事实源。
- 支线系统必须与现有主线/NPC/回响口径一致，不能突然切成另一套叙事语言。
- 正式任务系统若落地，必须以可配置、可兼容、可回退为目标，而不是一次性写死在 UI 逻辑里。

## 4. 回退条件

- 若新增规则让支线与主线边界失真，例如支线开始抢断主线播放，应回退。
- 若新增任务口径与当前项目叙事风格明显冲突，应回退。
- 若任务奖励设计被证明可重复刷取或破坏成长节奏，应回退。
- 若原著改写边界失守，出现不适配的露骨或敏感桥段，应回退。
