# 05. 内容接入接口

## 1. 目的

本文件定义支线任务系统的内容资产接入规则，确保任务原型、任务章节、任务 choice、奖励包和后续读取点都能以统一方式接入系统。

本文件还负责写清原著参考文本的改写边界，避免把离线素材直接搬进项目正文。

## 2. 一个任务章节最少要定义什么

### 2.1 一个合格的支线任务最少要定义什么

一个合格的支线任务至少要定义：

- `任务标题`
- `任务分类`
- `触发条件`
- `接取方式`
- `成功条件`
- `失败或错过条件`
- `奖励包`
- `至少一个后续读取点`

每条任务还必须说明：

- 它主要在补哪种旧账：旧案、旧情、师门、人情、利益还是退路。
- 它和哪条主线窗口产生交界。
- 它会写入哪些旗标、关系、路线或后续回响。
- 它是否属于唯一任务、限时任务或冲突组。

## 3. 一个合格 choice 至少要有哪三层差异

### 3.1 表层差异：任务承诺

- 玩家必须能理解这次是在补债、救人、封口、取证、换利益还是切割关系。

### 3.2 中层差异：任务成本

- 玩家至少要看到一种代价，例如灵石、人情、时间窗口、关系波动或潜在错过风险。

### 3.3 深层差异：任务后果

- choice 至少命中以下三项中的两项：
- 一项奖励包变化
- 一项关系/旗标/路线变化
- 一个后续读取点或后续线索变化

如果只有文案差异，没有任务后果差异，默认视为伪分支。

## 4. 何时必须写显式 `tradeoff`

- 任务是唯一奖励或限时任务时。
- 任务会让玩家在“旧情/旧债”和“资源/安全”之间明确取舍时。
- 任务会推动关系大幅变化时。
- 任务失败或错过会改变后续线索池时。
- 任务涉及师门站队、名声定性、暗线保留与否时。

## 5. 何时只允许走默认 `tradeoff`

- 常规回信、常规探路、常规交付这类低风险支线。
- 只是对既有路线人格做轻量确认的普通分支。
- 不涉及唯一奖励、不涉及长期错过、不改变后续线索池的低权重任务。

即便走默认 `tradeoff`，也仍要手写任务后果，不能只靠模板生成。

## 6. 什么是伪分支

- 按钮文案不同，但奖励、关系变化、读取点完全相同。
- 只改数值多少，不改后续承接。
- 只写“你处理了这件事”，不写这件事之后谁会记住、谁会失去、你以后还会不会再被这条线找上。
- 任务失败与任务错过完全没有差异。
- 写了原著梗，却没有落到当前项目的系统状态上。

## 7. 原著参考文本使用边界

- 离线参考素材路径：
- `D:\Program_python\game_xiuxian\Fan Ren Xiu Xian Chuan [ Geng Xin Zhi  Di - Wang Yu.txt`
- 该文件只用于：
- 校对人物关系骨架
- 校对阶段性事件顺序
- 提炼任务母题，例如“旧债”“师门试风向”“旧情未断”“残图余波”“幸存者后果”
- 该文件不得用于：
- 直接复制原文段落
- 直接搬运露骨性描写、性暴力、猥亵、猎艳桥段
- 直接作为运行时代码、测试脚本、配置表的数据源
- 直接替代当前项目中的主线事实

## 8. 原著改写规则

- 只保留事件骨架，不保留原文措辞。
- 只保留项目当前已承接的人物与线索，不强行引入首版无法承接的新角色资产。
- 将原著中的极端桥段改写为符合当前项目口径的“取证、护送、补债、清旧案、认旧情、断旧约、处理余波”。
- 让任务更像“你决定要不要把这件旧账接回手里”，而不是“你去完成一条剧情复刻清单”。

## 9. 作者最小接入清单

1. 这条任务属于哪一类。
2. 它为什么会在这个主线窗口出现。
3. 它的失败或错过条件是什么。
4. 它的奖励包是什么。
5. 它会写入哪些关系、旗标、路线或后续读取点。
6. 它是否会和同窗其他支线冲突。
7. 它是否通过伪分支检查。
