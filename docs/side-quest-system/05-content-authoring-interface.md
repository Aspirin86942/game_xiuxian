# 05. 内容接入接口

## 1. 目的

本文件定义地点委托系统的内容资产接入规则，确保委托原型、委托 choices、奖励预览和结果留痕都能以统一方式接入系统。

本文件同时规定原著参考素材的改写边界，避免把离线文本直接搬进项目正文。

## 2. 一个地点委托最少要定义什么

一个合格的地点委托至少要定义：

- `任务标题`
- `地点标签`
- `location`
- `境界门槛`
- `任务分类`
- `任务正文`
- `奖励预览`
- `choices`

每条委托还必须说明：

- 它为什么只能发生在这个地点。
- 它偏凡俗差使、山市买卖、宗门杂务，还是海上生计。
- 它办成后留下的是哪种轻量痕迹：资源、路线倾向、地方名声，还是一条结果记录。
- 若地点属于位置族，还必须说明它是绑定单点 `location`，还是复用 `visibleLocations` 覆盖整组地点。

## 2.1 玩家可见词面规则

-- `side quest / 支线任务` 不再作为玩家可见词面。
-- 系统总称统一为 `地点委托`。
-- 卡片状态默认映射为：
  - `available` -> `可接委托`
  - `active` -> `已接手`
  - `completed` -> `已办妥`
  - `failed` -> `已失手`
-- 接取动作统一为 `接下这笔委托`。

补充映射：

- 玩家在游历中先听到的是 `委托风声`，不是 `旧事线索`。
- 正式委托结果摘要继续使用 `已办妥 / 已失手`，不要回写成 `旧事了结 / 旧事错过`。
- `commission` 只映射为 `地点委托 / 委托风声 / 差使`；legacy 线索层才继续使用 `旧事 / 旧事线索`。

补充要求：

- 文案口径采用“半世界观化”，既不写系统后台说明，也不硬拗成难读的古白话。
- 若要表达奖励、关系、成本或结果，优先写成地方风声、差使、人情、买卖、去路与代价，而不是抽象成产品术语。
- 参考《凡人修仙传》时，只借母题与气质，不直接复制原句。

## 3. 一个合格 choice 至少要有哪三层差异

### 3.1 表层差异：任务承诺

- 玩家必须能理解这次是在求稳、求快、求利、求名声还是给自己留后手。

### 3.2 中层差异：任务成本

- 玩家至少要看到一种代价，例如灵石、物料、风险、地方名声或路线偏移。

### 3.3 深层差异：任务后果

- choice 至少命中以下三项中的两项：
  - 一项奖励包变化
  - 一项旗标 / 路线 / 地方评价变化
  - 一项结果摘要差异

如果只有文案差异，没有任务后果差异，默认视为伪分支。

补充要求：

- 玩家可见结果摘要不能直接写成“资源兑换”“关系后果”“情绪奖励”。
- 若要告诉玩家这笔委托留下了什么，必须写成世界里的结果：是赚到踏实、背走名声、欠下人情、摸清规矩，还是把后手先揣进袖里。

## 4. 何时必须写显式 `tradeoff`

- 委托是唯一奖励或唯一机会时
- 委托会让玩家在稳妥 / 逐利之间明确取舍时
- 委托会推动路线倾向明显变化时
- 委托失败会留下明确地方评价时
- 委托会消耗稀缺物料时

## 4.1 何时只允许走默认 `tradeoff`

- 低风险跑腿、常规探路、轻量买卖这类不改变系统边界的位置。
- 只是补一层地方风气，不改变结果态类型的选择。
- 不涉及唯一奖励、不涉及稀缺材料的普通分支。

## 5. 合法的地点委托约束

- 每条委托必须绑定唯一 `location`。
- 每条委托必须定义清楚 `minRealmScore / maxRealmScore`。
- 每条委托只能通过一次接取与一次结算收住。
- 每条委托的结果留痕必须轻量，不承担主线章节的解释职责。

作者不允许自行发明“委托做完解锁主线章节”之类新的系统职责。

## 7. 原著参考文本使用边界

- 离线参考素材：
  - 本机自备的《凡人修仙传》离线全文文本，路径不写入规则书
- 该文件只用于：
  - 校对人物关系骨架
  - 校对阶段性事件顺序
  - 提炼任务母题，例如旧债、旧案、旧情、守门、断尾
- 该文件不得用于：
  - 直接复制原文段落
  - 直接搬运露骨、敏感或不适配当前项目口径的桥段
  - 直接作为代码、测试脚本或配置表的数据源

## 8. 作者最小接入清单

1. 这条委托属于哪个地点。
2. 它为什么只能在这个地点发生。
3. 它对应哪个境界区间。
4. 它的失败条件或风险语义是什么。
5. 它的奖励预览是什么。
6. 它会写入哪些旗标、路线或结果摘要。
7. 它是否通过伪分支检查。
8. 它的玩家可见词面是否已按“地点委托 / 可接委托 / 已接手 / 已办妥 / 已失手 / 接下这笔委托”落地。

## 9. 设计底线

- 不允许只扩文本，不扩任务后果。
- 不允许只扩隐藏状态，不扩结果留痕。
- 不允许把明明属于地点生态的差使写成主线旧账尾巴。
- 不允许继续让地点委托承担主线卷末解释职责。
- 不允许把内部任务术语直接塞给玩家当界面标题或按钮文案。
